# Snooze (overflow action) — implementation plan

> Status: **Implemented** · 2026-06-12 · v1.5.0. Built as planned (decisions + as-built notes at the end).
> An extension of scheduled pins (see [2026-06-12-scheduled-pins.md](2026-06-12-scheduled-pins.md)) inside
> the editor-only Advanced feature family. Reuses the existing `pinAt` machinery — a snooze *is* a future
> `pinAt`.

## Goal

A **Snooze** action in the editor / popup overflow that hides a pinned note from the tray now and brings
it back later. Mechanically: **unpin now + set `pinAt = now + duration`**. The existing alarm + reconcile
re-pin it; the existing Advanced **"Pin at"** row is where the pending return is viewed, edited, or
cancelled (no separate Snooze control).

## Scope — locked decisions

- **Overflow action only.** A **Snooze** item in the shared `NoteEditorActions` overflow (`⋮`), so it
  appears in both the full editor and the quick-note popup. The **notification "Snooze" action is
  deferred** to a later slice.
- **Reuse `pinAt`.** No separate Snooze row in Advanced; a snooze writes `pinAt`, and the existing "Pin at"
  row shows / edits / clears it. "Un-snooze now" = re-pin (the stale future `pinAt` then clears, see below).
- **Gating:** the Snooze item is **hidden for new notes** and **shown-but-disabled for unpinned existing
  notes** (greyed, not removed); **enabled only when the note is pinned**.
- **Keep `unpinAt` on snooze.** Snooze preserves any existing auto-unpin window (it no-ops if the note is
  still unpinned when it fires).
- **Presets + a custom duration picker:** **30 minutes · 1 hour · 3 hours · 1 day · Custom…** (all
  durations, so each is simply `pinAt = now + Δ`). **Custom opens a duration picker** — *not* an absolute
  date/time — with **Days / Hours / Minutes** fields (**no seconds**), resolving to `pinAt = now + duration`.

## Behaviour change to scheduled pins (supersedes Q2/Q3)

The scheduled-pins slice decided manual pin/unpin leave the schedule untouched (timers no-op on fire). This
slice changes that so the UI stays truthful:

- **Pinning a note clears its `pinAt`** (and cancels that alarm) — a pinned note has no pending future pin.
- **Unpinning a note clears its `unpinAt`** (and cancels that alarm) — an unpinned note has no pending unpin.

These apply to `NotePinner.pin/pinAll` and `unpin/unpinAll` (so the list toggle, editor toggle, notification
Remove, etc. all clear consistently). **Snooze is the deliberate exception:** it unpins the *notification*
while **keeping `unpinAt`**, so it can't go through the normal `unpin()`.

Net effect on snooze coherence: re-pinning a snoozed note (un-snooze now) clears the pending `pinAt`
cleanly; letting it ride re-pins at `pinAt`; clearing "Pin at" cancels the return.

## The snooze operation

`NotePinner.snooze(noteId, untilMillis)`:
1. Load the note; if pinned, **raw-unpin** it — `setPinned(false)` + `controller.unpin(id)` — **without**
   clearing `unpinAt` (the snooze exception).
2. `setSchedule(pinAt = untilMillis, unpinAt = existing)`.
3. Arm the pin alarm (and keep/re-arm the unpin alarm if `unpinAt` is still future).
4. `reconcileService()` (the note left the pinned set).

Reuses `PinScheduler` and the reconcile model wholesale; no new alarm plumbing, no new permission.

## UI — the chooser

- **Overflow item** (`NoteEditorActions` → `NoteOverflowMenu`): "Snooze", shown when the note exists,
  `enabled = isPinned`. The chooser + duration dialog are hosted inside the shared `NoteOverflowMenu`, so
  both the editor and popup get them by passing a single `onSnooze(untilMillis)` callback.
- **Chooser form** (matching the time-picker form, as a duration picker): tapping "Snooze" opens a dialog
  listing the four presets + **Custom…**; **Custom…** switches the dialog to **Days / Hours / Minutes**
  number fields (validated > 0) → `until = now + duration`. A preset is just `now + Δ`.
- **Live action** (like pin/unpin/archive on an existing note): snooze takes effect immediately and updates
  `EditorUiState` (`isPinned = false`, `pinAt = until`) so the Advanced "Pin at" row reflects it. It does
  **not** close the editor/popup. Snooze input is a *duration*, but the stored/edited schedule stays the
  absolute `pinAt` shown in the "Pin at" row.

## Build order

1. **Clearing rules + `snooze` seam** — `NotePinner.pin/unpin` clear the matching time; `NotePinner.snooze`.
   Unit-tested over fakes (snooze unpins + sets `pinAt` + keeps `unpinAt`; pin clears `pinAt`; unpin clears
   `unpinAt`).
2. **Chooser UI** — presets + duration dialog; the `NoteEditorActions` overflow item (gated/disabled);
   `EditorViewModel.snooze`; wire editor + popup.
3. **Verify** — unit + a UI check that the disabled/hidden gating is correct and a snooze sets the Pin-at row.

## Invariants honoured

- No `INTERNET`, no new permission (inexact alarms, reused).
- `NotificationController` sole notification toucher; `PinScheduler` sole `AlarmManager` toucher.
- A note is never archived + pinned (snooze only acts on a pinned, non-archived note).
- Editor-only family; snooze rides the shared `NoteEditorActions`, so the popup gets it too (consistent),
  but it never appears for new notes.

## Testing

- **Unit (`NotePinnerTest`):** snooze raw-unpins + sets `pinAt` + **keeps `unpinAt`** + arms the alarm; pin
  clears `pinAt`; unpin clears `unpinAt`; re-pinning a snoozed note clears the pending `pinAt`.
- **Unit:** duration/preset → `until` computation (pure helper, incl. "Tomorrow morning").
- **Instrumented:** the overflow item is hidden for a new note and disabled for an unpinned existing note;
  snoozing a pinned note populates the Advanced "Pin at" row. Full duration-dialog interaction is manual.

## Risk / effort

**Small–medium.** The seam is tiny (reuses everything); the work is the chooser UI + the duration picker
(no built-in Material control) and threading the overflow action through the shared editor/popup row. The
clearing-rule change is small but touches a shared seam, so the unit tests around pin/unpin matter.

## Resolved decisions (2026-06-12)

1. **Granularity:** Days / Hours / Minutes — **no seconds** (below the inexact-alarm noise floor).
2. **Presets** are all durations: 30 min · 1 hour · 3 hours · 1 day · Custom… ("Tomorrow morning" dropped).
3. **Snooze does not close** the editor/popup (reversible state change).
4. **Chooser form:** a dialog matching the time-picker form — preset list, with Custom… switching to the
   Days/Hours/Minutes duration fields.

## Snooze for / until extension (2026-06-18)

The single "Snooze" overflow item was split into two:

- **"Snooze for..."** — the existing relative duration picker (presets + custom Days/Hours/Minutes).
- **"Snooze until..."** — an absolute date/time picker, reusing the `DateTimePickerDialog` from
  `ScheduleControls.kt` (changed from `private` to `internal`).

Both produce the same `onSnooze(untilMillis: Long)` callback — only the picker UI differs. Both items
are gated `enabled = isPinned`, matching the original. `DateTimePickerDialog` opens with `initialMillis`
and `earliestMillis` set to `System.currentTimeMillis()`. The string `action_snooze` was replaced by
`action_snooze_for` and `action_snooze_until`.
