# Stela — v1.1.0 Features Plan

> Three features rolled in after the v1.0.x personal-device testing pass. Companion to
> [2026-06-08-stela-design.md](2026-06-08-stela-design.md). Versioning: these are new
> features → **v1.1.0** (minor bump; `stelaVersionName` already at `1.1.0`).

## Slices & recommended order

- **A — Auto-capitalise Title & Description** *(done)* — keyboard capitalisation on the
  editor's two text fields. Trivial.
- **B — Swipe to unpin** *(next)* — opt-in setting: swiping a pinned notification unpins it
  instead of self-healing. Default off. Mirrors "Hide on lock screen".
- **C — Emoji per note** — a per-note emoji shown in the list + notification title via a
  derived `displayTitle`, with an emoji picker in the editor. Largest (schema migration +
  picker); do last.

---

## Slice A — Auto-capitalise Title & Description *(done)*

**Status (2026-06-10) — complete.** `KeyboardCapitalization.Sentences` added to both editor
text fields; build green. It's a soft-keyboard setting, so it's verified by build + a manual
device check (not JVM-testable).

**Confirmed decisions:**
- Both the Title and Description fields auto-capitalise via
  `KeyboardCapitalization.Sentences` (first letter, and after sentence punctuation).
- This sets the **soft keyboard only** — it does not force or transform the stored text
  (hardware keyboards / paste can still produce lowercase; that's intentional).

**What & where:** add `keyboardOptions = KeyboardOptions(capitalization =
KeyboardCapitalization.Sentences)` to both `OutlinedTextField`s in `EditorScreen`.

**Tasks:** 1. Add `keyboardOptions` to both fields. 2. Verify (build; manual: the keyboard
capitalises the first letter).

**Testing:** manual (keyboard behaviour isn't JVM-testable).

## Slice B — Swipe to unpin

**Confirmed decisions:**
- New **`swipeToUnpin`** boolean preference (DataStore), **default off**; a Settings toggle
  labelled **"Swipe to unpin"**, mirroring "Hide on lock screen".
- **Off (default):** current behaviour — the pinned notification is `setOngoing(true)` and
  its delete intent **re-asserts** (self-heals).
- **On:** the pinned notification is `setOngoing(false)` (so it is swipeable on Android <14;
  14+ already allows it) and its delete intent **unpins** the note (the existing unpin
  action). The note is **kept**, only unpinned.
- Toggling the setting **re-asserts** all pinned notes (re-post with the new behaviour),
  like the lock-screen setting.
- **Rename:** the notification **"Remove"** action becomes **"Unpin"** (it unpins rather
  than deletes). User-facing string `notification_action_remove` → "Unpin"; optionally
  rename the internal `ACTION_REMOVE`/`removeIntent` to match (touches
  `NotificationActionReceiverTest`).

**Units:** `Settings` + `SettingsRepository` gain `swipeToUnpin`; `SettingsViewModel` +
Settings UI toggle; `AndroidNotificationController` reads the flag (the way it reads
`hideOnLockScreen`) and branches `post()` on `setOngoing` + delete-intent target;
`NotePinner.reassertPinned` runs on toggle.

**Tasks:** 1. `swipeToUnpin` preference + settings mapping (TDD on the mapping). 2.
Controller branch (ongoing + delete intent) + flag wiring. 3. Settings toggle + reassert on
change. 4. Rename Remove → Unpin. 5. Verify (build, unit; manual/instrumented: on → swipe
unpins; off → self-heals).

**Testing:** JVM for the settings mapping; instrumented/manual for the swipe behaviour.

## Slice C — Emoji per note

**Confirmed decisions:**
- A new **`emoji`** column on `Note` (keep `iconId` as-is; the notification **small icon
  stays the silhouette** — a separate concept). One emoji per note, optional and clearable.
- **`displayTitle` is derived, not user-edited:** `emoji + " " + title` when an emoji is
  set, else just `title` (**no default emoji**). Used in the **note-list row**, the
  **pinned-notification title**, and as the **share** subject/text.
- **Picker: the official `androidx.emoji2:emoji2-emojipicker`** `EmojiPickerView`, hosted in
  a bottom sheet / dialog via `AndroidView` (categories, search, skin tones for one new
  dependency).
- The emoji **supersedes** the deferred v2 custom-icon-set picker (`iconId` stays a
  reserved, display-unused column). Users may still type emojis directly into Title or
  Description if they want.

**Units:**
- **Data:** `Note.emoji` (empty default) + a **Room migration** (export the new schema; add
  a migration test). `NoteRepository.create`/`update` carry the emoji.
- **Derive:** a pure, unit-testable `displayTitle` helper (on `Note` or a small util).
- **Editor:** `EditorUiState` / `EditorViewModel` gain the emoji; `EditorScreen` adds an
  emoji affordance (show current emoji + pick/clear) that opens the picker sheet.
- **Consumers:** the list-row headline, `AndroidNotificationController` pinned title, and
  `shareNote` all switch to `displayTitle`.

**Tasks:** 1. `Note.emoji` + Room migration (+ migration test) + repository plumbing. 2.
`displayTitle` helper (TDD). 3. Editor emoji picker (`emoji2-emojipicker` dependency +
bottom sheet). 4. Route list + notification + share through `displayTitle`. 5. Verify
(build, unit, migration test; manual: pick/clear emoji → shows in list, notification, and
share).

**Testing:** JVM for `displayTitle` and the repository; instrumented for the Room migration
and the picker flow.

---

## Sequencing

**A (next) → B → C.** A is a quick win; B mirrors the existing lock-screen setting pattern;
C is the largest — a schema migration and a new dependency — so it's done last and with the
most care.
