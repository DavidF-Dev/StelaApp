## Decisions

- **v1 distribution: GitHub Releases** ‚ÄĒ ship a signed APK attached to a tagged release.
- **Future channels (not v1):** may also publish to **Google Play** (AAB + store listing +
  Data Safety form + privacy-policy URL) and/or **F-Droid** (builds from source, signs
  itself; needs `fastlane/metadata`). See "Future channels" below.
- **Version:** `versionName = "1.0.0"`, `versionCode = 1`.
- **R8:** code shrinking + resource shrinking **enabled** for release; smoke-tested. No
  custom ProGuard rules needed (Room/Compose/DataStore ship consumer rules). Release APK
  ‚Čą 1.3 MB.
- **Signing:** release builds read a **git-ignored `keystore.properties`**; when it is
  absent (CI/fresh clone) the release falls back to debug signing so the build stays
  runnable for testing.

## Signing setup

`keystore.properties` (git-ignored; copy from `keystore.properties.template`):

```
storeFile=/absolute/path/to/stela-release.jks
storePassword=‚Ä¶
keyAlias=stela
keyPassword=‚Ä¶
```

`app/build.gradle.kts` loads it and selects the signing config; the keystore and
properties file are never committed (`.gitignore` already covers `*.jks`,
`*.keystore`, `keystore.properties`).

### Generate the signing key

```
keytool -genkeypair -v -keystore stela-release.jks -alias stela \
  -keyalg RSA -keysize 2048 -validity 10000
```

- `-validity 10000` (~27 years) must outlive every future update.
- It prompts for a keystore password, a key password, and a distinguished name (your
  name/org ‚ÄĒ any truthful value is fine for self-distribution).

### Signing best practices

- **Back up the keystore and both passwords** somewhere durable (password manager /
  encrypted backup). **If the key is lost, you can no longer ship updates** that existing
  users can install ‚ÄĒ they would have to uninstall and reinstall.
- **Reuse the same key for every update.** Android rejects an update whose signature does
  not match the installed app.
- **Never commit** the keystore or `keystore.properties`.
- Publish the certificate **SHA-256 fingerprint** (`keytool -list -v -keystore
  stela-release.jks`) in the release notes so users can verify the APK's authenticity.
- **Cross-channel caveat:** an APK self-signed for GitHub has a *different* signature from
  an F-Droid build (F-Droid signs its own) and from a Play build under Play App Signing.
  Users cannot switch channels via in-place update; pick a canonical channel per install.

## Release steps (v1.0.0)

1. Ensure `keystore.properties` exists and the key is generated (above).
2. Build the signed APK:
   ```
   ./gradlew assembleRelease
   ```
   Output: `app/build/outputs/apk/release/app-release.apk`.
3. Install on a device and smoke-test (release ‚Č† debug: R8 is active).
4. Commit, then **tag and push**:
   ```
   git tag -a v1.0.0 -m "Stela v1.0.0"
   git push origin v1.0.0
   ```
5. Create the GitHub Release for `v1.0.0`, paste the `CHANGELOG.md` v1 section as the
   body, attach `app-release.apk`, and note the certificate SHA-256 fingerprint.
