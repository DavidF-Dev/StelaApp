# Stela — v1.0.0 Release Readiness

> How v1 is built, signed, and shipped. Companion to the README (user-facing) and
> [2026-06-08-phase6-polish.md](2026-06-08-phase6-polish.md) (feature work).

## Decisions

- **v1 distribution: GitHub Releases** — ship a signed APK attached to a tagged release.
- **Future channels (not v1):** may also publish to **Google Play** (AAB + store listing +
  Data Safety form + privacy-policy URL) and/or **F-Droid** (builds from source, signs
  itself; needs `fastlane/metadata`). See "Future channels" below.
- **Version:** `versionName = "1.0.0"`, `versionCode = 1`.
- **R8:** code shrinking + resource shrinking **enabled** for release; smoke-tested. No
  custom ProGuard rules needed (Room/Compose/DataStore ship consumer rules). Release APK
  ≈ 1.3 MB.
- **Signing:** release builds read a **git-ignored `keystore.properties`**; when it is
  absent (CI/fresh clone) the release falls back to debug signing so the build stays
  runnable for testing.

## Signing setup

`keystore.properties` (git-ignored; copy from `keystore.properties.template`):

```
storeFile=/absolute/path/to/stela-release.jks
storePassword=…
keyAlias=stela
keyPassword=…
```

`app/build.gradle.kts` loads it and selects the signing config; the keystore and
properties file are never committed (`.gitignore` already covers `*.jks`,
`*.keystore`, `keystore.properties`).

### Generate the signing key

```
keytool -genkeypair -v -keystore stela-release.jks -alias stela \
  -keyalg RSA -keysize 2048 -validity 10000
```

- `-validity 10000` (~27 years) must outlive every future update.
- It prompts for a keystore password, a key password, and a distinguished name (your
  name/org — any truthful value is fine for self-distribution).

### Signing best practices

- **Back up the keystore and both passwords** somewhere durable (password manager /
  encrypted backup). **If the key is lost, you can no longer ship updates** that existing
  users can install — they would have to uninstall and reinstall.
- **Reuse the same key for every update.** Android rejects an update whose signature does
  not match the installed app.
- **Never commit** the keystore or `keystore.properties`.
- Publish the certificate **SHA-256 fingerprint** (`keytool -list -v -keystore
  stela-release.jks`) in the release notes so users can verify the APK's authenticity.
- **Cross-channel caveat:** an APK self-signed for GitHub has a *different* signature from
  an F-Droid build (F-Droid signs its own) and from a Play build under Play App Signing.
  Users cannot switch channels via in-place update; pick a canonical channel per install.

## Release steps (v1.0.0)

1. Ensure `keystore.properties` exists and the key is generated (above).
2. Build the signed APK:
   ```
   ./gradlew assembleRelease
   ```
   Output: `app/build/outputs/apk/release/app-release.apk`.
3. Install on a device and smoke-test (release ≠ debug: R8 is active).
4. Commit, then **tag and push**:
   ```
   git tag -a v1.0.0 -m "Stela v1.0.0"
   git push origin v1.0.0
   ```
5. Create the GitHub Release for `v1.0.0`, paste the `CHANGELOG.md` v1 section as the
   body, attach `app-release.apk`, and note the certificate SHA-256 fingerprint.

## Release automation (planned — Option B: local script)

A local `scripts/release.ps1` (**not yet implemented**) will make a release one command:
bump `versionName` and write the `CHANGELOG.md` section, then run the script. It builds and
signs **locally** (the key never leaves the machine), extracts the changelog, and creates
the GitHub Release with the APK attached. Requires `gh` (authenticated) and a present
`keystore.properties`.

**Decisions:**
- **Tagging:** the script lets `gh release create v$version` create the tag at HEAD (tag
  creation is not done separately).
- **versionCode:** *derived from `versionName`* in `build.gradle.kts` —
  `major*10000 + minor*100 + patch` (e.g. `1.0.0` → `10000`, `1.2.3` → `10203`). One source
  of truth (only `versionName` is edited), monotonic with semver, no separate field to
  forget. Replaces the current literal `versionCode = 1` when implemented.
- **Changelog extraction:** read the `## [x.y.z]` section from `CHANGELOG.md` (Keep a
  Changelog format) — everything from the version's heading up to the next `## [` heading.
  Matches on the bracketed version, so the date / "unreleased" suffix is ignored. The
  heading line itself is excluded (the release title already shows the version).
- **Fingerprint:** the signing-cert SHA-256 is constant across releases, so it is appended
  from a fixed value (optionally recomputed via `apksigner verify --print-certs`).
- **Guard rails (abort before building):**
  - working tree is dirty (uncommitted changes),
  - `keystore.properties` is missing,
  - the `CHANGELOG.md` section for the version is empty or missing,
  - a `v$version` tag or release already exists.

**Flow:** check guard rails → `assembleRelease` → extract notes + fingerprint →
`gh release create v$version app-release.apk --title … --notes-file …`.

**Still manual (inherent):** bumping `versionName` and authoring the changelog section;
everything else (versionCode, build, sign, changelog extraction, tag, release + upload) is
automated.

## Quality gates (verified 2026-06-09)

- `assembleRelease` with R8 — green; release APK ≈ 1.3 MB.
- Release-build smoke test on the emulator (create → pin → notification) — passes.
- 54 JVM unit + 21 instrumented tests — green.
- `lintDebug` — no fatal issues. Remaining warnings are informational: ~35
  `GradleDependency` (newer versions available) + AGP version, plus minor
  `QueryPermissionsNeeded` (best-effort OEM-autostart `resolveActivity`; the row is hidden
  gracefully when unresolved) and `ObsoleteSdkInt` (the `mipmap-anydpi-v26` qualifier is
  cosmetically redundant at minSdk 26 but kept — `mipmap-anydpi` failed resource linking).

## Future channels (when chosen)

  store listing (title, descriptions, screenshots, 512 icon, feature graphic), a content
  rating, a **Data Safety** declaration ("no data collected" — true; no `INTERNET`), a
  privacy-policy URL, and a `specialUse` foreground-service justification in the Console
  (already drafted in the manifest property).
- **F-Droid:** add `fastlane/metadata/android/…` (descriptions, changelogs, graphics) and
  submit to `fdroiddata`; F-Droid builds from source and signs the result.
